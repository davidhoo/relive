# SQLite 可行性评估

> 评估 SQLite 是否满足 Relive 项目需求
> 对比方案：SQLite vs PostgreSQL

---

## 一、需求分析

### 1.1 数据规模

| 指标 | 预估值 | 说明 |
|------|--------|------|
| **照片数量** | 11 万张（初始） | 每年新增约 3000 张 |
| **数据库大小** | ~500MB - 1GB | 元数据 + EXIF + 分析结果 |
| **单条记录** | ~5-10KB | 包含完整 EXIF JSON |
| **5年后规模** | ~12.5 万张 | 增长可控 |
| **10年后规模** | ~14 万张 | 增长缓慢 |

### 1.2 并发需求

| 场景 | 并发特征 | 频率 |
|------|---------|------|
| **照片扫描** | 写入密集 | 首次全量 + 日常增量 |
| **AI 分析** | 更新密集 | 每天 500-1000 次 |
| **ESP32 请求** | 读取 | 每天 1-10 次 |
| **Web 浏览** | 读取 | 低频（个人使用） |
| **统计查询** | 读取 | 低频 |

**并发特点**：
- ✅ **读多写少**：90% 读取，10% 写入
- ✅ **单用户为主**：个人/家庭使用，不是多用户系统
- ✅ **写入可串行**：照片扫描可以排队处理
- ✅ **读取压力小**：ESP32 和 Web 访问频率低

### 1.3 查询需求

| 查询类型 | 复杂度 | 频率 |
|---------|--------|------|
| **往年今日查询** | 中等 | 每天 1-10 次 |
| **按评分排序** | 简单 | 每次请求 |
| **按城市筛选** | 简单 | 低频 |
| **按分类筛选** | 简单 | 低频 |
| **全文搜索** | 中等 | 低频 |
| **统计分析** | 中等 | 低频 |

### 1.4 部署环境

- ✅ **群晖 NAS**：资源有限，不适合重量级数据库
- ✅ **Docker 容器**：需要轻量级方案
- ✅ **备份简单**：单文件备份更方便
- ✅ **维护成本**：希望零维护

## 二、SQLite 评估

### 2.1 优势 ✅

#### 性能优势
- ✅ **读取性能优秀**：
  - 单表查询速度快（< 10ms）
  - 11万条记录轻松处理
  - 适合读多写少场景

- ✅ **嵌入式部署**：
  - 无需独立数据库服务
  - 无需维护和管理
  - 启动速度快

- ✅ **资源占用低**：
  - 内存占用小（~数MB）
  - CPU 占用低
  - 适合 NAS 环境

#### 功能优势
- ✅ **完整的 SQL 支持**：
  - 支持复杂查询
  - 支持事务（ACID）
  - 支持索引优化
  - 支持 JSON 字段（SQLite 3.38+）

- ✅ **单文件存储**：
  - 备份简单（复制文件即可）
  - 迁移方便
  - 版本控制友好

- ✅ **跨平台**：
  - 所有平台支持
  - Golang 驱动成熟（mattn/go-sqlite3）

#### 开发优势
- ✅ **零配置**：
  - 无需安装数据库服务
  - 无需配置连接池
  - 开发调试方便

- ✅ **可靠性高**：
  - 久经考验（20+ 年）
  - 广泛使用（Android、iOS、浏览器）
  - Bug 少，稳定性好

### 2.2 限制 ⚠️

#### 并发限制
- ⚠️ **写入并发**：
  - 同时只能有一个写入事务
  - 多个读取可以并行
  - **影响**：对 Relive 影响小（写入可串行）

- ⚠️ **锁机制**：
  - 数据库级别锁（非表级）
  - 写入时会阻塞读取（短暂）
  - **影响**：对 Relive 影响小（读取压力低）

#### 性能限制
- ⚠️ **数据库大小**：
  - 推荐 < 1GB（性能最佳）
  - 实际支持到 TB 级别
  - **评估**：Relive 预计 500MB-1GB ✅

- ⚠️ **复杂查询**：
  - JOIN 性能不如 PostgreSQL
  - 全文搜索功能较弱
  - **影响**：Relive 查询简单，影响小

#### 功能限制
- ⚠️ **用户管理**：
  - 无内置用户权限系统
  - **影响**：Relive 单用户，无影响

- ⚠️ **复制/集群**：
  - 不支持主从复制
  - **影响**：Relive 单机部署，无需复制

### 2.3 性能测试数据

**参考数据**（类似规模项目）：
```
数据量：100,000 条记录
单条记录：~5KB（含 JSON）
数据库大小：~500MB

查询性能：
- 按 ID 查询：< 1ms
- 按日期范围查询（索引）：< 10ms
- 按评分排序（TOP 10）：< 20ms
- 全表扫描：~100-200ms

写入性能：
- 单条插入：< 5ms
- 批量插入（100条）：< 50ms
- 事务提交：< 10ms
```

**结论**：性能完全满足 Relive 需求 ✅

## 三、PostgreSQL 对比

### 3.1 PostgreSQL 优势

- ✅ **并发性能**：支持高并发读写
- ✅ **复杂查询**：JOIN、子查询性能更好
- ✅ **全文搜索**：内置强大的全文搜索
- ✅ **扩展性**：支持复制、分片
- ✅ **JSON 支持**：更强大的 JSON 查询

### 3.2 PostgreSQL 劣势

- ❌ **资源占用**：内存占用 ~50-100MB+
- ❌ **维护成本**：需要独立服务，配置复杂
- ❌ **备份复杂**：需要 pg_dump
- ❌ **部署复杂**：Docker 多容器
- ❌ **过度设计**：Relive 用不到高级功能

### 3.3 对比总结

| 维度 | SQLite | PostgreSQL | Relive 需求 |
|------|--------|------------|-------------|
| **数据规模** | < 1GB（推荐） | GB-TB | ~500MB ✅ |
| **并发读** | 优秀 | 优秀 | 低频 ✅ |
| **并发写** | 单写 | 高并发 | 可串行 ✅ |
| **查询复杂度** | 简单-中等 | 复杂 | 简单 ✅ |
| **全文搜索** | 基础（FTS5） | 强大 | 基础足够 ✅ |
| **资源占用** | 极低 | 中等 | 希望低 ✅ |
| **维护成本** | 零维护 | 需要维护 | 希望零 ✅ |
| **备份** | 复制文件 | pg_dump | 希望简单 ✅ |
| **部署** | 嵌入式 | 独立服务 | 希望简单 ✅ |
| **扩展性** | 有限 | 优秀 | 无需扩展 ✅ |

## 四、结论和建议

### 4.1 结论：SQLite 完全满足需求 ✅

**理由**：
1. ✅ **数据规模**：11万张照片，预计 500MB-1GB，完全在 SQLite 舒适区
2. ✅ **并发模式**：读多写少，单用户，SQLite 的并发限制不是问题
3. ✅ **查询需求**：简单查询为主，SQLite 性能完全够用
4. ✅ **部署环境**：群晖 NAS，资源有限，SQLite 轻量级更合适
5. ✅ **维护成本**：个人项目，SQLite 零维护更友好
6. ✅ **备份迁移**：单文件，备份和迁移极其简单

**参考案例**：
- InkTime 项目也使用 SQLite ✅
- 支持数万张照片，运行稳定
- 备份简单，迁移方便

### 4.2 建议方案

#### 方案 A：纯 SQLite（推荐）⭐

**适用场景**：
- 个人/家庭使用
- 照片数量 < 50 万张
- 单机部署

**优点**：
- ✅ 部署简单（单文件）
- ✅ 零维护成本
- ✅ 备份方便（复制文件）
- ✅ 资源占用低

**缺点**：
- ⚠️ 扩展性有限（但 Relive 不需要）

#### 方案 B：SQLite + 未来可迁移 PostgreSQL

**设计原则**：
- 使用标准 SQL（兼容两者）
- 使用 ORM（GORM）抽象数据库
- 避免使用特定数据库特性

**迁移策略**：
```
当满足以下条件时考虑迁移到 PostgreSQL：
1. 照片数量 > 50 万张
2. 多用户并发访问
3. 需要复杂的全文搜索
4. 需要数据复制/备份
```

**迁移成本**：
- 使用 ORM，迁移成本低
- 主要是配置变更
- 数据导出/导入

### 4.3 最终推荐

**使用 SQLite** ✅

**原因**：
1. 完全满足当前需求
2. 简化部署和维护
3. 降低资源占用
4. 参考项目验证可行
5. 未来可平滑迁移（如需要）

**性能优化措施**：
- ✅ 合理设计索引
- ✅ 使用 WAL 模式（Write-Ahead Logging）
- ✅ 批量操作使用事务
- ✅ 合理设置缓存大小
- ✅ 定期 VACUUM 清理

## 五、SQLite 配置建议

### 5.1 推荐配置

```go
import (
    "database/sql"
    _ "github.com/mattn/go-sqlite3"
)

// 连接字符串
dsn := "file:relive.db?cache=shared&mode=rwc&_journal_mode=WAL&_busy_timeout=5000&_synchronous=NORMAL&_cache_size=10000"

// 参数说明：
// - cache=shared: 共享缓存模式
// - mode=rwc: 读写创建模式
// - _journal_mode=WAL: WAL 模式（更好的并发）
// - _busy_timeout=5000: 锁超时 5 秒
// - _synchronous=NORMAL: 正常同步（性能与安全平衡）
// - _cache_size=10000: 缓存大小 10000 页（约 40MB）
```

### 5.2 性能优化

```sql
-- WAL 模式（推荐）
PRAGMA journal_mode = WAL;

-- 设置缓存大小（10000 页 = 约 40MB）
PRAGMA cache_size = -40000;

-- 正常同步模式（性能与安全平衡）
PRAGMA synchronous = NORMAL;

-- 启用外键约束
PRAGMA foreign_keys = ON;

-- 设置临时目录（避免跨设备）
PRAGMA temp_store = MEMORY;

-- 自动清理（定期）
PRAGMA auto_vacuum = INCREMENTAL;
```

### 5.3 维护策略

```bash
# 定期优化（每月）
sqlite3 relive.db "VACUUM;"

# 分析统计信息（优化查询）
sqlite3 relive.db "ANALYZE;"

# 检查完整性（每周）
sqlite3 relive.db "PRAGMA integrity_check;"

# 备份（每天）
cp relive.db relive.db.backup.$(date +%Y%m%d)
```
