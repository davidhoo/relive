# People Clustering Cross-Photo Create Guard Design

**Goal:** 减少多人合照导致的误并，同时保留“单张新脸可以附着到已有 Person”的能力。

**Scope:** `backend/internal/service/people_service.go` 与对应测试；本次不修改数据模型、不新增配置项、不改变现有 attach 阈值。

## Current Problem

当前增量聚类流程会对所有 `pending` 人脸统一建图，再对连通分量做两类决策：

1. 能附着到已有 `Person` 则整组附着
2. 不能附着但组件大小 `>= min_cluster_faces` 则新建 `Person`

这带来一个明显问题：

- 同一张照片里的多张脸可以彼此连边
- 如果这些脸之间 embedding 足够接近，它们会形成一个组件
- 即使这些脸都只来自同一张照片，系统也会把这个组件当作“有足够证据的新人物”

这会把多人合照中的不同人误并成一个新 `Person`。

## Alternatives Considered

### Option 1: 完全禁止同图脸连边

优点：
- 能最直接阻断同图误并

缺点：
- 冷启动能力下降过多
- 同图脸无法共享局部结构信息
- 对已有人物附着也更保守

### Option 2: 只在“新建人物”路径上要求跨照片证据

优点：
- 保留现有附着到已有人物的能力
- 只收紧误并风险最高的“创建新人”分支
- 改动小，便于验证

缺点：
- 同图人脸仍然会出现在同一组件中
- 组件内部结构仍可能放大某些边界样本

### Option 3: 引入更复杂的密度/权重规则

优点：
- 理论上可更细致地区分误并与漏并

缺点：
- 超出本轮边界
- 需要重新设计聚类打分，不适合作为补丁修复

**Decision:** 采用 Option 2。

## Design

### Rule Change

保留现有图构建与附着逻辑：

- `pending` 人脸仍然照常建图
- 组件仍然优先尝试附着到已有 `Person`

仅修改“创建新人物”的前提：

- 组件不能附着到已有 `Person`
- 且组件大小 `>= min_cluster_faces`
- 且组件至少覆盖 `2` 张不同照片

若组件只来自 `1` 张照片，则即使组件大小达到 `min_cluster_faces`，也不允许新建人物，而是保持 `pending`。

### Why This Boundary

“创建新人物”是误并成本最高的决策：

- 一旦错误新建或错误合并，后续会影响原型、人脸统计和展示分类
- 同图多脸最不可靠，因为它们共享拍摄条件、角度和场景，容易出现局部相似但并非同人的情况

要求跨照片证据后，系统会更倾向于：

- 同图组件先等待
- 直到后续其他照片提供支持，再创建新人物

### Non-Goals

本次不做以下改动：

- 不禁止同图脸与已有 `Person` 原型的附着
- 不修改 `link_threshold` / `attach_threshold` / `min_cluster_faces`
- 不对已有 `assigned` 老人物做批量重聚类
- 不新增数据库字段或后台任务

## Testing Strategy

需要新增三类回归测试：

1. 同图两张新脸，不能附着到已有 `Person` 时，不创建新人物而保持 `pending`
2. 同图两张新脸，若能附着到已有 `Person`，仍然允许 `assigned`
3. 跨两张照片的两张新脸，不能附着到已有 `Person` 时，仍然允许创建新人物

## Expected Outcome

这个修复的净效果是：

- **误并下降**：多人合照不再靠单张照片内部互相支持就新建人物
- **召回略降**：某些只在一张照片里同时出现的陌生人，会先停留在 `pending`
- **已有人物附着能力保持**：单张新脸仍可并入已有 `Person`
