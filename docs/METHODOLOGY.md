# Relive 项目开发方法论

## 文档驱动开发 + Claude Code 最佳实践

> 本文档定义了 Relive 项目的开发方法和工作流程

---

## 一、核心理念

### 1.1 文档驱动开发（DDD）

**定义**：先写文档，再写代码。文档是唯一的真理来源（Single Source of Truth）。

**核心原则**：
- 📝 **文档先行**：任何功能开发前，先在文档中定义清楚
- 🎯 **明确需求**：通过文档迭代，消除理解偏差
- 📚 **知识沉淀**：文档即项目知识库
- 🔄 **持续更新**：代码变化，文档同步更新

### 1.2 Claude Code + Skill 的优势

- 🤖 **AI 理解力强**：可以深度理解文档上下文
- 🛠️ **自动化能力**：Skill 可以封装复杂流程
- 📖 **文档优先友好**：AI 特别擅长基于文档生成代码
- 🔁 **快速迭代**：需求变化时，改文档即可

## 二、项目文档体系

### 2.1 文档分层

```
docs/
├── REQUIREMENTS.md          # 需求文档（核心）⭐
├── ARCHITECTURE.md          # 架构设计文档
├── API_DESIGN.md            # API 接口设计
├── DATABASE_SCHEMA.md       # 数据库设计
├── DEVICE_PROTOCOL.md       # 设备通信协议
├── SCORING_ALGORITHM.md     # 评分算法详细设计
├── DEVELOPMENT_GUIDE.md     # 开发指南
└── CHANGELOG.md             # 变更日志
```

### 2.2 文档优先级（开发顺序）

**Phase 1：需求阶段** ✅ 当前
1. `REQUIREMENTS.md` - 需求文档（核心，必须最先完成）

**Phase 2：设计阶段**
2. `DATABASE_SCHEMA.md` - 数据库设计（数据是基础）
3. `API_DESIGN.md` - API 设计（定义接口契约）
4. `ARCHITECTURE.md` - 架构设计（技术方案）

**Phase 3：实现阶段**
5. `SCORING_ALGORITHM.md` - 算法详细设计
6. `DEVICE_PROTOCOL.md` - 硬件通信协议
7. `DEVELOPMENT_GUIDE.md` - 开发规范

**Phase 4：迭代阶段**
8. `CHANGELOG.md` - 持续更新

## 三、文档驱动的工作流程

### 3.1 标准开发流程

```
┌─────────────────────────────────────────────────┐
│ 1. 需求讨论（对话）                              │
│    - 通过问答明确需求细节                        │
│    - 记录到 REQUIREMENTS.md                     │
└──────────────┬──────────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────────┐
│ 2. 设计文档编写                                  │
│    - 根据需求编写设计文档                        │
│    - 数据库、API、架构设计                       │
└──────────────┬──────────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────────┐
│ 3. 设计评审（对话）                              │
│    - 用户审阅设计文档                            │
│    - 讨论和修改设计                              │
└──────────────┬──────────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────────┐
│ 4. 使用 Skill 生成代码                           │
│    - 基于文档自动生成代码框架                    │
│    - 保持代码和文档一致                          │
└──────────────┬──────────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────────┐
│ 5. 代码评审和优化（使用 /simplify）             │
│    - 审查生成的代码                              │
│    - 优化和重构                                  │
└──────────────┬──────────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────────┐
│ 6. 测试和验证                                    │
│    - 功能测试                                    │
│    - 更新文档（如有偏差）                        │
└──────────────┬──────────────────────────────────┘
               │
               ▼
┌─────────────────────────────────────────────────┐
│ 7. 提交代码（使用 /commit）                     │
│    - 提交符合规范的 commit                       │
│    - 引用相关文档                                │
└─────────────────────────────────────────────────┘
```

### 3.2 需求变更流程

```
变更请求 → 更新 REQUIREMENTS.md → 评估影响范围
→ 更新相关设计文档 → 修改代码 → 测试 → 提交
```

**关键原则**：文档先改，代码后改

## 四、充分利用 Claude Code Skill

### 4.1 当前可用的 Skill

#### `/simplify`
**用途**：审查代码质量、重构、优化
**使用场景**：
- 代码生成后的质量检查
- 发现代码重复或过度设计
- 优化性能和可读性

**使用时机**：
- 完成一个功能模块后
- 合并代码前
- 代码评审阶段

#### `/commit`（系统内置）
**用途**：智能生成 commit message
**使用场景**：
- 代码提交时自动生成规范的 commit
- 引用相关文档或 issue

### 4.2 自定义 Skill 建议（未来可扩展）

我们可以为 Relive 项目创建专属 Skill：

#### `/design` - 生成设计文档
```
根据 REQUIREMENTS.md 自动生成：
- API 设计文档
- 数据库 schema
- 架构图
```

#### `/scaffold` - 生成代码框架
```
基于设计文档生成：
- 目录结构
- 空的类和函数定义
- 测试文件模板
```

#### `/analyze` - 分析照片测试
```
调用 Qwen API 分析指定照片
用于测试和调试 AI 分析功能
```

#### `/doc-sync` - 文档代码同步检查
```
检查代码和文档是否一致
发现偏差并提示更新
```

## 五、文档编写规范

### 5.1 需求文档（REQUIREMENTS.md）

**结构**：
```markdown
# 一、项目概述
# 二、核心使用场景
# 三、功能需求
# 四、待明确的需求（用 ❓ 标记）
# 五、非功能需求
# 六、技术约束
# 七、开放问题清单
```

**编写原则**：
- ✅ 使用明确的语言（避免"可能"、"也许"）
- ✅ 用户视角（描述用户能做什么，而非技术实现）
- ✅ 可验证（需求完成后可以测试验证）
- ✅ 标记状态（待明确/已确认/已实现）

### 5.2 设计文档

**必须包含**：
- 设计目标和原则
- 详细的技术方案
- 接口定义（输入输出）
- 数据结构
- 流程图（如需要）
- 示例代码片段
- 异常处理
- 性能考虑

### 5.3 文档元信息

每个文档开头应包含：
```markdown
> 状态：草稿 / 评审中 / 已确认 / 已实现
> 负责人：[如果多人协作]
> 最后更新：2026-02-28
> 依赖文档：REQUIREMENTS.md
```

## 七、协作建议

### 7.1 与 Claude Code 的协作模式

**高效对话方式**：
```
❌ 错误："帮我写个照片扫描功能"
✅ 正确："根据 REQUIREMENTS.md 第 3.1 节和 DATABASE_SCHEMA.md，
         实现照片扫描服务，需要支持增量扫描和 EXIF 提取"
```

**文档引用**：
- 明确引用文档章节
- Claude Code 会自动读取相关文档
- 保持上下文一致

### 7.2 文档更新纪律

**原则**：代码和文档同步更新

**实践**：
- 每次需求变更，先更新 REQUIREMENTS.md
- 每次 API 变更，同步更新 API_DESIGN.md
- 重大架构变更，更新 ARCHITECTURE.md
- 记录到 CHANGELOG.md

## 八、成功标准

### 项目成功标准
- ✅ 文档完整准确（可以直接交给新人理解项目）
- ✅ 代码和文档一致（文档是代码的准确描述）
- ✅ 需求可追溯（每个功能都能在文档中找到出处）
- ✅ 易于维护（文档更新带动代码更新）

### 文档质量标准
- ✅ 无歧义（两个人看到的理解一致）
- ✅ 完整性（覆盖所有核心场景）
- ✅ 可验证（可以根据文档验收功能）
- ✅ 易读性（新人可以快速上手）

## 九、常见陷阱和避免方法

### ❌ 陷阱 1：文档和代码脱节
**表现**：文档写完就不管，代码按自己想法写
**避免**：代码变化必须同步更新文档

### ❌ 陷阱 2：过度设计
**表现**：文档写得太细，约束了实现
**避免**：需求文档描述"做什么"，设计文档描述"怎么做"，分层清晰

### ❌ 陷阱 3：文档形式主义
**表现**：为了文档而文档，不能指导实际开发
**避免**：文档必须是"可执行"的，能直接生成代码

### ❌ 陷阱 4：跳过文档直接写代码
**表现**：觉得文档浪费时间，直接开写
**避免**：记住"慢就是快"，前期文档投入会节省后期返工时间
